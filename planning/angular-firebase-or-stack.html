Angular+Firebase OR Stack

Task ID: PR-04 · Date: 2026-05-27 (rev. visual identity) · Upstream deps: PR-01, PR-02 · Baseline: PR-02, PR-03 · Themes: T1–T6

Browser-first operations-research stack for the Ukwazi portfolio: Angular 21 LTS, Firebase Hosting, read-only Storage for synthetic seeds, and Web Workers (or Cloud Functions only when the browser budget is exceeded). No Firebase Auth — public demos with scenario compare/save via in-session state, IndexedDB, and JSON export. Visual identity is distinct from integraldecision.com: dark minimal chrome with subtle gradients (reference: PHNX Foundry topology demo). Demos use one of two presentation modes — a four-panel simulation workbench for live, parameter-driven runs (reference: Aircraft Stability Visualiser), or a vertical notebook layout for narrative, step-through studies (reference: Observable portfolio). The home gallery lists both types with the same card chrome. Visualization is limited to two libraries: D3 for custom simulation canvases and ECharts (ngx-echarts) for standard charts. Heavy mine-planning MIP and proprietary Whittle/Deswik integration remain out of scope.

Visual identity & UX

Design direction

  • Palette — Near-black background (#0c0c0e), elevated panels in charcoal greys, off-white body text, one cool accent (steel blue / muted cyan) for links, active controls, and chart series. Subtle radial gradients on hero and panel headers only — not full glassmorphism.
  • Typography — System UI stack; tabular figures for KPIs and OR coefficients; no Material Design or Ant Design defaults.
  • Component shell — Spartan UI or Zard UI (copy-paste, Tailwind v4) for buttons, sliders, tabs, and dialogs. Own the theme tokens in-repo; do not match integraldecision green or ecosystem.Ai Workbench chrome.
  • Gallery index — Home route lists runnable demos as cards (title, one-line pitch, T1–T6 badge, presentation-mode pill: workbench or notebook, “Open demo →”). Observable gallery pattern — each card links to a live artifact, not a tutorial site.
  • Method context — Workbench: collapsible explainer in panel D (PHNX “What to Watch”). Notebook: prose cells interleaved with widgets. No separate Learn route hierarchy.

Presentation modes — pick per demo

Not every project is a live simulation. Choose the mode that fits the OR story; both share the same dark theme, D3/ECharts stack, and gallery entry. Set presentationMode in demos.json; the router lazy-loads either WorkbenchLayoutComponent or NotebookLayoutComponent under the same /demo/:demoId route.

Interactive demos that need a persistent control surface and live response. Responsive 2×2 grid on desktop; stacked on narrow viewports (controls → sim → charts → metrics).

Panel B follows the Aircraft Stability split (controls + live response + diagnostic charts). Panel D follows PHNX Foundry’s iteration counter, material/compliance metrics, and explainer accordion. Side-by-side scenario compare (topology demo “Compare Side-by-Side”) is a stretch overlay on panel B/C, not a separate app shell.

Notebook layout notebook

Narrative demos scroll vertically through ordered cells — Observable-style, but implemented as Angular components (not iframe embeds). Each cell is one concern; reactive inputs in an upstream cell re-run downstream outputs without full-page reload.

Markdown cell — context, formulas, mining scenario setup, citations
Input cell — period selector, scenario picker, threshold sliders, CSV seed toggle
Chart cell — ngx-echarts waterfall, SPC, Gantt, KPI strip (full width)
Map / canvas cell — Leaflet bench heatmap or D3 figure when spatial story matters
Insight cell — computed variance table, alert callouts, export JSON / snapshot

Reconciliation dashboards (T3) fit naturally: prose frames the accountability question → waterfall quantifies the gap → map locates the bench → insight cell states who owns the variance. Pit schedule reviews (T1) fit as Gantt + bench map cells stepping through precedence and period KPIs without cramming everything into four fixed quadrants.

Cross-cutting platform choices

  • Angular 21 LTS standalone — feature folders per demo with lazy loadComponent routes; zoneless change detection; shared WorkbenchLayoutComponent or NotebookLayoutComponent selected from catalog metadata.
  • Firebase — Hosting (SPA) and Storage (read-only seeds). No Auth or user-scoped Firestore in MVP. Optional Cloud Functions (Python OR backend) + App Check only if a stretch demo exceeds browser budget.
  • State — Signals-first UI; scenario snapshots in IndexedDB or in-session compare; export/import JSON; large result arrays in Worker memory or IndexedDB, not remote documents.
  • Web Workers — One pool sized to navigator.hardwareConcurrency − 1; pass large arrays with transferables; terminate idle Workers on route leave.
  • Testing — Vitest for OR kernels; Playwright smoke on hosted demo URLs for interview links.

Anti-patterns

Do not run large MIP / pit-shell optimisation in the browser main thread.

Life-of-mine nested-shell or schedule MIP with thousands of binary variables will freeze the tab and fail on mobile. If a demo needs real MIP scale, pre-compute offline and visualise results — or call a Cloud Function with a timeout and progress UI. Browser LP/IP is limited to teaching-scale models (hundreds of variables, short solve times in a Worker).

Do not pretend Whittle, Deswik, or Vulcan integration.

Reviewers at a planning consultancy will spot fake logos or “import .whittle” claims. Use synthetic block models (PR-07b), CSV/GeoJSON upload, and explicit labels: “toy pit shell”, “synthetic schedule”, “method illustration only”.

  • Chart library sprawl — adding Chart.js or ngx-charts alongside ECharts splits theming and doubles bundle size; stay on D3 + ECharts only.
  • Main-thread DES at scale — thousands of haul cycles block UI; always Worker + capped horizon.
  • Firestore as compute cache — storing full DES event logs or MIP constraint matrices blows document limits; use Storage or client IndexedDB.
  • Solver keys in client — remote MIP APIs belong in Functions with secrets.
  • One monolith app — without lazy routes, T2 DES + T3 maps + T5 MDP in one bundle hurts load time.
  • Gym / hospital demos as headline — PR-03 gap analysis: re-skin entities, lead with mining KPIs.
  • Sign-in for public demos — use local persistence or JSON export instead of uid-scoped Firestore.
  • Wrong presentation mode — forcing a reconciliation narrative or schedule post-mortem into four fixed quadrants when a notebook scroll tells the story better; forcing DES/MDP live sims into static notebook cells when a workbench is expected.
  • integraldecision look-alike — separate repo must read as mining OR portfolio, not decision-science tutorial site.
