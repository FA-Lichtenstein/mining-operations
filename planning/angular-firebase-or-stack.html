Angular+Firebase OR Stack

Task ID: PR-04 · Date: 2026-05-27 · Upstream deps: PR-01, PR-02 · Baseline: PR-02, PR-03 · Themes: T1–T6

This document defines a browser-first operations-research stack for Ukwazi portfolio demos: Angular 17+ with Firebase Hosting, Auth, and Firestore, plus Web Workers or Cloud Functions for anything heavier than toy LP/IP or small DES/MDP runs. Heavy mine-planning MIP and proprietary Whittle/Deswik integration are explicitly out of scope — demos use synthetic CSV/GeoJSON, open solvers, and honest “method showcase” framing. Portable module patterns are audited from integraldecision.com and ecosystem.Ai Angular Workbench experience cited on the CV.

Cross-cutting platform choices

  • Angular 17+ standalone — feature folders per demo (features/haulage-des/, features/reconciliation-dash/) with lazy loadComponent routes; heavier than integraldecision’s current eager main bundle, but same route taxonomy.
  • Firebase — Hosting (SPA), Auth, Firestore (metadata + user scenarios), Storage (CSV/GeoJSON), Functions (optional Python OR backend), App Check on public endpoints. Security Rules enforce uid-scoped writes; sensitive OR logic stays in Functions.
  • State — Signals-first UI (Angular 17+; zoneless viable on greenfield); RxJS via AsyncPipe or toSignal() for Firestore streams; immutable scenario snapshots in Firestore; large result arrays in IndexedDB or Worker memory, not Firestore documents.
  • Web Workers — One pool sized to navigator.hardwareConcurrency − 1; pass large arrays with transferables to avoid copy overhead; terminate idle Workers on route leave.
  • Testing — Vitest/Jest for OR kernels; Playwright smoke on hosted demo URLs for interview links.

Anti-patterns

Do not run large MIP / pit-shell optimisation in the browser main thread.

Life-of-mine nested-shell or schedule MIP with thousands of binary variables will freeze the tab and fail on mobile. If a demo needs real MIP scale, pre-compute offline and visualise results — or call a Cloud Function with a timeout and progress UI. Browser LP/IP is limited to teaching-scale models (hundreds of variables, short solve times in a Worker).

Do not pretend Whittle, Deswik, or Vulcan integration.

Reviewers at a planning consultancy will spot fake logos or “import .whittle” claims. Use synthetic block models (PR-07b), CSV/GeoJSON upload, and explicit labels: “toy pit shell”, “synthetic schedule”, “method illustration only”. Reference commercial tools in landscape research (PR-08) without implying this portfolio connects to them.

  • Main-thread DES at scale — thousands of haul cycles block UI; always Worker + capped horizon.
  • Firestore as compute cache — storing full DES event logs or MIP constraint matrices blows document limits; use Storage or client IndexedDB.
  • Solver keys in client — remote MIP APIs belong in Functions with secrets.
  • One monolith app — without lazy routes, T2 DES + T3 maps + T5 MDP in one bundle hurts load time; mirror integraldecision routes but lazy-load each OR feature.
  • Gym / hospital demos as headline — PR-03 gap analysis: re-skin entities, lead with mining KPIs.

Suggested Angular+Firebase sketch

Monorepo-style Angular app on Firebase Hosting with routes /learn (demo catalog with T1–T6 tags), /learn/:demoId (tutorial), /decide/:demoId (interactive run). Lazy-load each demo folder. Firestore collections: scenarios/{uid}/{demoId} (saved parameters + summary KPIs), catalog/demos (metadata for cards-menu). Auth: public read for Learn; Google sign-in to save Decide runs. T2 flagship: haulage-des Worker module, Chart.js time-series, JSON seed from PR-07b. T3: Leaflet + GeoJSON variance layer. T5: fleet-mdp Worker with schema-driven form (ecosystem.Ai Workbench pattern). T4 optional: WSM Decide fork of integraldecision weightedsummodel. Cloud Function stub solveLp only if PR-19 selects a larger IP demo — not default.
